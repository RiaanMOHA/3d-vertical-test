## Architecture and design notes

### The inlining transformation

Before:
```html
<script src="https://cdnjs.cloudflare.com/ajax/libs/three.js/r128/three.min.js"></script>
```

After:
```html
<script>
/* Three.js r128 — inlined so file works offline/Quick Look/email preview */
/* ...603,445 bytes of minified Three.js r128... */
</script>
```

Everything else in the file is unchanged. File grew from 35,141 → 639,855 bytes (~625 KB). The rest of the file (DOCTYPE, HTML structure, CSS, the main scene-building `<script>` block, chip UI, orbit + first-person controls) is untouched and still matches the architecture described in the prior handoff.

### File protection strategy

`chflags uchg` (macOS user-immutable flag) was chosen over `chmod 444` because:
- `chmod 444` stops modification but NOT deletion (a process with write access to the parent directory can still `rm` a 444 file).
- `chflags uchg` blocks both modification AND deletion, even by root, until explicitly cleared.
- No sudo required for user-owned files.
- Unlock is one command: `chflags nouchg <path>`.

Verified flag is set via `ls -lO` (capital O — macOS-specific flag that shows file flags). Output includes `uchg` in the flags column when active.

### Why not kill the other Claude sessions?

User declined explicitly ("no, find a way so that it doest get deleted or make it react"). The `chflags` approach is strictly better for a non-dev user because:
- It doesn't require the user to understand PIDs or `kill` commands.
- It doesn't interrupt their other VS Code work.
- It's a single atomic change with a clear undo.

### Inlined Three.js vs. CDN

For an occasionally-shared 3D demo, inlining is worth it even though it didn't solve the preview-pane issue here. Benefits:
- File works offline.
- File works in environments with restrictive network (corporate firewalls, air-gapped machines, some countries).
- No dependency on cdnjs surviving or the r128 URL remaining valid.

Downside:
- File grew 18× (35KB → 625KB). Still reasonable for a local HTML but noticeably larger in chat / email.
- License compliance: inlined content retains the MIT license header at the top of the `three.min.js` source. That's correct and sufficient for Three.js.

### Recipient-side workarounds

For a shareable 3D demo, hosting beats file-sending every time. Tier list:
1. **Netlify Drop** (https://app.netlify.com/drop) — drag, get URL, instant. No account strictly required. Works with single HTML file.
2. **Vercel** — similar, slightly more friction if no CLI installed.
3. **GitHub Pages** — if user has a repo already. Needs git knowledge.
4. **File send + "open in browser" instruction** — works, but requires recipient cooperation.
5. **File send with no instruction** — fails (what happened this session).

Next time shared 3D HTML is on the table, lead with option 1.
