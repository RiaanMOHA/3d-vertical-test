## Problems encountered and how they were solved

**Problem 1: Initial pano analysis was geometrically impossible.**
Symptom: Reported layout put bed (2m) + closet (1.4m) + frosted window on a single 2.7m wall. User caught this immediately.
Root cause: Subagent conflated "visible together in a wide pano" with "on the same wall". a/6 showed AC + frosted + closet in one field-of-view but those features span two adjacent walls, not one.
Fix: Pulled the blueprint PDF, measured dims (2.7×2.7), and reconstructed the layout from per-wall feature evidence. Closet is on south wall (accessing 物入 storage behind), frosted window is on west wall with AC, bed long side runs along west wall.
Lesson: When multiple "long" features seem to be on the same wall, check wall length against feature lengths. If they don't fit, they're on adjacent walls you're seeing at an angle.

**Problem 2: Port 5175 was already in use.**
Symptom: `npm run dev` with `port: 5175, strictPort: true` → "Port 5175 is already in use".
Root cause: An orphaned vite process from the deleted `room-1-3d-v3/` folder was still running (PID 53764, `/Users/riaan/3d-vertical-test/room-1-3d-v3/node_modules/.bin/vite`).
Fix: Changed port to 5176 in `vite.config.js` and updated the chip URL in `ozu-test.html` to match. Left the orphan process alone.

**Problem 3: Headless WebGL context creation failed.**
Symptom: `THREE.WebGLRenderer: A WebGL context could not be created. ... ErrorMessage = BindToCurrentSequence failed`.
Root cause: Default headless Chrome doesn't enable WebGL. `--use-gl=swiftshader` alone isn't enough on macOS.
Fix: Switched to `headless: 'new'` plus the flag combo `--enable-unsafe-swiftshader --use-gl=angle --use-angle=swiftshader --enable-webgl --ignore-gpu-blocklist`.

**Problem 4: Open door panel invisible in the straight-on entrance shot.**
Symptom: shot-entrance.png showed the doorway (black rect) and bookshelf but no door panel jutting into the room.
Root cause: Camera FOV cropped it. Door panel is at x=[1.9, 2.7], z=[2.66, 2.7]. From camera at (1.35, 1.6, 1.35) facing +x, the panel center is ~55° to the right. Vertical FOV 70°, horizontal FOV ≈ 96° at 1280×800 aspect → half-horizontal is ~48°, so the panel sits just outside the visible frame.
Fix: Added a diagonal orbit shot from (0.3, 1.6, 0.3) looking toward (2.6, 1.0, 2.6) that captured it. Also temporarily recolored the panel red to prove it was rendering in the expected place, then reverted.

**Problem 5: The user told me multiple times the analysis was slow.**
Symptom: "so slow", "slowww", "why so slo11111".
Root cause: Reading panos one at a time via Read tool is bandwidth-heavy, and I was over-deliberating wall orientations mid-read.
Fix first time: Delegated to an Explore subagent to batch-read remaining panos. But the subagent also returned a wrong answer.
Fix second time: For screenshot diagnostics, skipped the iterative debug-by-rerun loop in favor of accepting the MVP result.
Lesson: When a task involves many mechanical image reads or many shell roundtrips, batch early. Don't bounce between analysis and IO.

## Architecture and design notes

**Coordinate convention per room-N-3d project:**
Each room chooses its own origin corner and axes. Room-2 v1:
- Origin at corner-ac-window on floor.
- +x along window-wall toward corner-entrance-window.
- +z along ac-wall toward corner-ac-closet.
- +y up. Right-handed.

Room-1 v4 uses origin at corner-cabinet-entrance, +x along entrance-wall, +z along cabinet-wall. So the conventions are NOT uniform across rooms — each room's origin is the corner alphabetically first in its own wall-pair set, and axes follow the two walls meeting at that corner in some consistent way.

When building future rooms, pick a convention that lets the bed sit simply (long side along an axis-aligned wall, head at origin) and document it at the top of constants.js.

**Door swing pattern (generalized from room-1):**
Hinge at the corner where entrance-wall meets the "closet/cabinet" wall. Door opens 90° inward so the panel ends up parallel to the closet/cabinet wall. Exact rotation sign depends on which side of the origin the hinge is on.

For a hinge at (W, 0, D) swinging open along wall z=D: panel local geometry at x=[0, T], z=[-panelW, 0], group rotation.y = +π/2.

For a hinge at (0, 0, 0) swinging open along wall x=0 (room-1 convention): panel local geometry at x=[0, panelW], z=[-T, 0], group rotation.y = -π/2.

The pattern is: local +x and +z always correspond to (hinge→thickness) and (hinge→free-edge-along-wall) respectively, and the rotation sign flips to send the free edge inward.

**v1 simplifications from room-1 v4:**
- No textures; flat MeshStandardMaterial throughout.
- Bed head/footboard = solid aabb boxes (no bars or rails) — room-1 v4's metalFrame function iterates along panel.z which fails for room-2's bed orientation (thin z dimension).
- Bookshelf = gray box with tier marker strips on front face (no posts, no X-bracing, no real tiers).
- Desk = flat top + 4 legs (no PBR dark-wood top).
- AC = plain white box (no vents or detail).
- Pendant unchanged from v4 — 4 emissive shades on a wood frame.

These are all intentional MVP choices per user preference "ship MVP early, don't over-polish".
