## Architecture and design notes

### Per-property pattern
Each "property" (`test`, `ozu-1-test`, future ones) follows this shape:
- **Exterior page** at port 8080 (e.g. `ozu-test.html`) — inline three.js scene + chips. This is the **only** entry point. Chips link out to interior builds.
- **Interior build(s)** as separate Vite projects on dedicated ports. Each is interior-only. Reachable only via chip click; direct URL access redirects to the exterior page (using `?from=outside` query param).
- **`← outside` link** in interior pages routes back to the exterior. **`projects` link** routes to root `index.html`.

### Coordinate system in `ozu-house-3d`
- X = east-west (0 to 6.30, west to east)
- Z = north-south (0 to 7.20, north to south, matching blueprint orientation)
- Y = up (0 = floor, +2.40 = 1F ceiling, +4.80 will be 2F ceiling)

### floor1.js wall-with-gaps helpers
Two helpers in `ozu-house-3d/src/floors/floor1.js`:
- `wallX(group, z, x_a, x_b, gaps=[], height=H)` — wall along the X axis at fixed z, with optional `[[g0, g1], ...]` gap ranges (door/window openings).
- `wallZ(group, x, z_a, z_b, gaps=[], height=H)` — wall along the Z axis at fixed x, same gap convention.
Both call `makeWall` which builds a `THREE.BoxGeometry` and rotates it by `atan2(dz, dx)`. Use this helper pattern for Phase 3's 2F walls so the code stays uniform.

### Room paint approach
LDK is one big rectangle painted first; genkan is a smaller rectangle painted on top with `y += 0.001` to avoid Z-fighting. This is a low-effort way to mark sub-zones inside a larger room without geometry subtraction. Reuse in 2F if a similar pattern arises (none expected — 2F rooms are cleanly separated).
