### Phase 12 — Variant switcher

Goal: one URL, three chips, click to switch between v2 / v3-rect / v3-walls.

Approach: URL query string drives static-imported variant selection. `?v=v2` loads v2, `?v=v3-rect` loads v3-rect, no param defaults to v3-walls. Page reloads on chip click (simpler than runtime scene swapping).

**File restructure (single bash call):**
- `mkdir -p src/variants/{v2,v3-rect,v3-walls}`
- `git mv src/{constants,room,furniture}.js src/variants/v3-walls/` (current files become the v3-walls variant)
- `git show main:src/constants.js > src/variants/v2/constants.js` (and same for room/furniture — recovered v2 from main branch via git show)
- `cp ../../room-1-3d-v3/src/{constants,room,furniture}.js src/variants/v3-rect/` (copied from sibling folder)

This cleanly preserved all 3 codebases — v2, v3-rect, v3-walls — in one project.

**Variant index.js (3 files):** Each `src/variants/X/index.js` re-exports `ROOM`, `CAMERA`, a `LABEL` string, and a `buildScene(parent)` function that calls the variant's `buildRoom + buildFurniture`. Templated identically across the three.

**main.js rewrite:** Used `Write`. Top of file:
```js
import * as v2       from './variants/v2/index.js';
import * as v3rect   from './variants/v3-rect/index.js';
import * as v3walls  from './variants/v3-walls/index.js';

const VARIANTS = { 'v2': v2, 'v3-rect': v3rect, 'v3-walls': v3walls };
const DEFAULT_VARIANT = 'v3-walls';
const urlVar = new URLSearchParams(location.search).get('v');
const activeName = VARIANTS[urlVar] ? urlVar : DEFAULT_VARIANT;
const variant = VARIANTS[activeName];
const ROOM   = variant.ROOM;
const CAMERA = variant.CAMERA;
```

Added DOM updates to mark active chip and show label. Replaced `buildRoom(scene); buildFurniture(scene);` with `variant.buildScene(scene);`. Rest of file (renderer, lighting, controls, walk/orbit, animation loop) unchanged.

**index.html updates:** Added CSS for `#variant-chips` (top-center flex with backdrop-blur chips) and `.variant-chip.active` style (rgba 0.2 white background, bold). Added `<span id="variant-label">` to the mode-line. Added the chip HTML:

## Learnings and discoveries

### About variant patterns in this project

- "Variant" is the user's preferred mental model for trying things. NOT iterating in place. Always create a new file/folder/branch when exploring an alternative.
- Variants can be: sibling docs (`room-map.md` + `room-map-photos.md`), sibling folders (`room-1-3d/` + `room-1-3d-v3/`), git branches (`main` + `room-1-v3-wall-fixes`), or scene modules within one app (`src/variants/{v2, v3-rect, v3-walls}/`).
- The user values being able to compare side-by-side. The variant switcher (one URL, three chips) is the cleanest realization of this.

### About the room photos vs room-map.md

- Photos beat room-map.md for: bed orientation, exact furniture positions.
- room-map.md is correct about: wall adjacencies (cabinet ↔ ac opposite, entrance ↔ window opposite), corner naming (alphabetical), photo folder structure (per-corner).
- room-map.md's "bed long side against ac-wall" is **wrong**. Photos show bed along window-wall with head at corner-cabinet-window.
- room-map.md's room dimensions (2.7×2.7×2.4) are assumed, not measured. Furniture inferences (bed 1.95m + desk 1.2m + shelf 0.4m = 3.55m) suggest the room may actually be 2.7×3.6m (the rectangular 6帖 layout).

### About the v3-walls coord system (user's Phase 9 spec)

- New coord system: cabinet-wall at x=0, entrance-wall at z=0, window-wall at x=W, ac-wall at z=D. Opposites: cabinet ↔ window, entrance ↔ ac.
- This is a **different wall topology** than v2/v3-rect (which had cabinet ↔ ac opposite, matching room-map.md).
- New corner names that follow alphabetically: corner-cabinet-entrance (0,0), corner-entrance-window (W,0), corner-ac-cabinet (0,D), corner-ac-window (W,D).
- The v3-walls spec did NOT change BED constants — bed mattress is still at x=0.7..2.7, z=1.7..2.7. Under the new wall topology, this places the bed along ac-wall (z=D) again. So v3-walls bed is in the same physical position as v2; only the wall rendering and feature mapping changed.

### About git + npm projects

- Vite 8 + three.js 0.184 + symlinks in `public/` work fine for cross-project HTML linking.
- `git mv` preserves git history of moves (shows as "renamed" in commit). Used this when moving constants/room/furniture into variants/v3-walls/.
- `git show <branch>:<path>` extracts file content from another branch without checking out. Used to copy v2 files into variants/v2/ from the main branch.
- GitHub accepts ~750MB initial pushes without complaint. Took less than a minute over a normal connection.

### About user preferences (this session, reinforced or new)

- "Plain language, no jargon" — confirmed multiple times. When I gave a technical answer about ports + git checkout, user responded "huh???".
- "Make a variant, not REPLACE IT" — strong, recurring preference. When in doubt, create a new thing instead of editing the old one.
- "yessssssssss" with elongated letters = mild exasperation that I asked permission instead of just doing the obvious next thing. When user says they want X and the next step is obviously Y, do Y without asking.
- The "delete first" preference (from prior memory) doesn't apply when a variant approach is available — variants ARE the alternative to delete-first.
- For visual/UI things, the user prefers fewer URLs to remember. Single-URL with chips beats multiple URLs.
- User does NOT want commit history obsession; "just push it" was the right read of "all" + "same name as this project + public".

### Variant switcher mechanism

URL query string drives variant selection at page load. No runtime swapping (page reloads on chip click). This is intentionally simple — avoids needing dispose() logic, scene re-attachment, camera re-targeting in JS.

The chip elements are plain `<a href="?v=X">` tags. Clicking navigates, browser reloads, main.js reads the new `?v=` value, imports use ESM static imports of all three variants but only the active one's `buildScene` is called.

ESM static imports of all 3 variants means Vite bundles all three at build time. Dev hot-reload works on individual variant files (any edit to `variants/X/constants.js` reloads the page).

The HUD shows the active chip (rgba 0.2 white background, bold) and a label inline with the mode display.

Camera config is pulled from the active variant's CAMERA export. Variants can have different orbit targets, walk start positions, and look-at points. The mode toggle (M / ESC) and walk controls (WASD) all use these per-variant values.

### File layout pattern for a Vite three.js multi-variant project

```
project/
├── index.html              ← HUD + canvas + variant chips
├── package.json            ← deps + scripts
├── public/                 ← Vite-served static assets (e.g. cross-project symlinks)
└── src/
    ├── main.js             ← renderer, lighting, controls; variant dispatch
    └── variants/
        └── <name>/
            ├── constants.js   ← ROOM, COLOR, furniture dims, CAMERA
            ├── room.js        ← buildRoom(parent): walls + features
            ├── furniture.js   ← buildFurniture(parent): bed, desk, etc.
            └── index.js       ← re-exports ROOM, CAMERA + LABEL + buildScene(parent)
```

Each variant is self-contained (no shared imports between variants). Trades duplication for independence — variants can diverge freely without affecting each other.
