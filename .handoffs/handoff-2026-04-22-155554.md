## Learnings and discoveries

### About room-map.md as source of truth

- `/Users/riaan/3d-vertical-test/ozu-test/interior-images/room-1/room-map.md` is a user-written spec that MUST be followed strictly. It establishes the wall-feature-name taxonomy that this project uses going forward.

## Architecture and design notes

### Project layout decision: separate Vite project per room vs unified HTML

Current state: exterior lives in `ozu-test.html` (single-file three.js). room-1 lives in `room-1-3d/` Vite project. They link to each other via HUD chips. This gives each "scene" its own tooling but requires the Vite dev server to be running for the exterior → room-1 link to work.

Alternatives considered: (a) port exterior into the Vite project and add a chip-based scene switcher (like ozu-test.html has); (b) put all rooms back into the single ozu-test.html file. User chose "just link them" → current state.

**When to rethink:** if/when more rooms are built, consider unifying under one Vite project with a scene registry pattern (similar to ozu-test.html's `registerScene('id', cfg)`). Avoids symlink gymnastics and makes the dev server dependency less fragile.

### Wall rendering approach in room-1 v2

Per user spec, walls are **thin boxes (thickness 0.05m)** positioned OUTSIDE the room volume — inner face of each wall at the room coordinate (x=0, x=W, z=0, z=D). This way the interior volume is clean from 0..W on x and 0..D on z.

Wall FEATURES (windows, doors, closet door, AC, coat hooks) are **flat colored rectangles offset by BIAS=0.003m from the wall surface** to avoid z-fighting. No real cutouts. This is explicitly a v1 simplification — later versions may punch real holes (in v1 this is what caused the room-1 v1 rejection partially; user wanted simpler).

The AC unit is a real 3D box (0.8×0.25×0.2m) that protrudes INTO the room from ac-wall. It's the one wall feature that's not a flat rect.

### Furniture rendering approach

Every piece is composed of axis-aligned boxes (BoxGeometry) with flat colors. No rotation, no complex geometry. A helper `aabb(r, color, extra)` takes `{x1,x2,z1,z2,y1,y2}` and expands to a correctly-positioned box. Another helper `boxMesh(w,h,d,cx,cy,cz,color,extra)` builds from size+center.

Shelf and chair legs use explicit loops over corner positions. Pendant uses a cross of two perpendicular boxes at ceiling + 4 emissive cylinders hanging at shade positions.

### Lighting recipe

- `AmbientLight(0xffffff, 0.6)` — even fill
- `DirectionalLight(0xffffff, 0.4)` from (5,8,3) targeting room center — subtle shading
- Pendant shades: emissive `MeshStandardMaterial` with emissive color + intensity 0.7 — glows but doesn't illuminate

**No point lights or spot lights** — prior v1 had them and user said they "blew out everything". Ambient-first is the rule going forward for interior rooms.

### Controls architecture

Two modes, both present, toggled by `M`:
- **orbit**: three-addons `OrbitControls`. Target (1.35, 1.0, 1.35), initial position (4.5, 3.5, 4.5). Damped. Useful for top-down verification.
- **walk**: custom click-drag look + WASD translate. NOT pointer-locked. Uses Euler YXZ on the camera. `enterWalk()` uses `camera.lookAt(walkLookAt)` + `camera.rotation.reorder('YXZ')` to seed walkYaw/walkPitch. No collision (per spec).

Walk is the default on load. ESC exits walk back to orbit (so user sees the outside box — this is a minor wart).

### Cross-project link pattern

Vite project `public/ozu-test.html` is a **symlink** to `../../ozu-test.html`. Vite serves it at `/ozu-test.html`. The room-1-3d HUD links to that path. The exterior's HUD links back to `http://localhost:5173/` directly.

### Coordinate system vs room-map.md ASCII diagram

room-map.md's ASCII diagram has cabinet-wall at the TOP. User's locked spec has cabinet-wall at z=0 (low z). In a top-down orbit view with +x right and +z down-on-screen, cabinet-wall appears at the TOP of the screen and ac-wall at the BOTTOM — matching ASCII. User's top-down verification checklist phrased it the opposite way ("bed along top edge = ac-wall") which is how the screen looks when viewed with +z DOWN (the conventional three.js top-down view has camera at positive y, looking down, and by default +z goes DOWN on screen making ac-wall appear at bottom). The verification script works mathematically regardless of render orientation.

Key invariant: **wall positions on real coordinates are**:
- entrance-wall: x=0
- window-wall: x=W=2.7
- cabinet-wall: z=0
- ac-wall: z=D=2.7
