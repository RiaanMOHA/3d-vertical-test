## Learnings and discoveries

### Texture pipeline

- Polyhaven's free CC0 CDN URL pattern `https://dl.polyhaven.com/file/ph-assets/Textures/jpg/1k/{slug}/{slug}_{suffix}_1k.jpg` works reliably for `wood_floor`, `painted_plaster_wall`, `dark_wood`. All verified this session.
- `curl -fsSL` is the right flag combo: `-f` fails on HTTP errors (exits non-zero), `-s` silent, `-S` show errors on failure, `-L` follow redirects.
- Per-material texture cloning is cheap and necessary: if 4 shelf tiers share one material with `repeat.set(0.4, 0.4)`, each tier's face repeats the same way. Fine for identical tiles. But when different faces want different repeats (desk top vs shelf tier), the `pbrMaterial` helper that clones per call is the right abstraction.
- `TextureLoader.load()` returns immediately (async). Scene may render briefly with untextured materials until textures arrive. Three.js handles the update automatically via `needsUpdate` + the material's internal refresh.
- `BoxGeometry` wraps textures around all 6 faces, but for walls we only see the inside face. Tiling applied via `repeat.set(length/sourceSize, height/sourceSize)` gives a realistic scale on the visible face; the near-invisible edge faces look mildly warped but acceptable.

### Door rotation math

- For a door hinged at a corner (world origin), wrapped in a `THREE.Group` with `rotation.y = -π/2`:
  - Rotation maps local (x, y, z) → world (-z, y, x).
  - Place door panel so local x = door length direction, local z = door thickness. Panel at local z ∈ [-thickness, 0] so that post-rotation, world x ∈ [0, thickness] (stays inside the room, doesn't poke through the adjacent wall).
  - Handle sits on the door's room-facing face when closed, which is local +z (the room side pre-rotation). Wait, careful: in this setup the door is ALWAYS open, so the closed-state face doesn't matter. What matters is where the handle ends up AFTER rotation. Handle at local z = -thickness - hD/2 (further "back" than panel) ends up at world x = thickness + hD/2 (protrudes into room on +x side).
- This math implies a slight compromise: a real door pivots around a jamb that's inset from the wall; a door hinged exactly at a wall corner will either poke through the adjacent wall when open or be outside the room when closed. We pick the former visual (door flush with cabinet-wall when open).

### Variant switcher mechanism (unchanged from prior sessions, noted for completeness)

URL query string drives static ESM import at page load:
```js
import * as v1 from './variants/v1/index.js';
import * as v2 from './variants/v2/index.js';
import * as v3 from './variants/v3/index.js';
import * as v4 from './variants/v4/index.js';
const VARIANTS = { v1, v2, v3, v4 };
const activeName = VARIANTS[new URLSearchParams(location.search).get('v')] ? ... : 'v4';
const variant = VARIANTS[activeName];
variant.buildScene(scene);
```

Each variant's `index.js` exports `ROOM`, `CAMERA`, `LABEL`, and a `buildScene(parent)` function that calls its variant-local `buildRoom + buildFurniture`. Shared code (renderer, lighting, walk/orbit controls, animation loop) lives in `main.js`.

### Texture pipeline

```
pbrMaterial('wood_floor', u, v, extras)
  ├── loadSet('wood_floor')          (cached per slug)
  │     ├── TextureLoader.load(.../wood_floor_diff.jpg)   → set color to sRGB
  │     ├── TextureLoader.load(.../wood_floor_nor_gl.jpg)
  │     └── TextureLoader.load(.../wood_floor_rough.jpg)
  ├── clone(map)         (per-caller clone so each mat has own repeat)
  ├── clone(normalMap)
  ├── clone(roughnessMap)
  └── apply repeat.set(u, v) → new MeshStandardMaterial({ map, normalMap, roughnessMap, ...extras })
```

Texture files live under `room-1-3d/public/textures/{slug}/`, accessible via Vite at `http://localhost:5174/textures/{slug}/{slug}_{map}.jpg`. Only used by v4; v1/v2/v3 use flat colors.
