## Problems encountered and how they were solved

### 1. Duplicate `const clock` SyntaxError (Phase 1)
- **Symptom:** Whole scene renders dark after switching away from exterior. No errors visible to user.
- **Root cause:** Two identical `const clock = new THREE.Clock();` declarations on adjacent lines inside the IIFE — SyntaxError at parse time, so the entire script never runs. No `switchScene('exterior')` call, no render loop.
- **Fix:** Deleted one line.
- **Avoid next time:** When the UI does nothing at all (not even init), suspect a parse error in the IIFE. Check devtools console for SyntaxError. The symptom is "nothing" rather than "wrong thing".

### 2. Misinterpreting a/b/c/d subfolders as separate rooms (Phases 2–5, 8)
- **Symptom:** Built wildly wrong layouts for room-2 (bedroom + study suite, then split+suite variants) and room-1 (wrong room size, wrong bed orientation).
- **Root cause:** Assumed the 4 subfolders represented 4 different physical rooms. They're actually 4 different **furniture stagings of the same architectural room**. Same walls, windows, AC, lamp fixture.
- **Tell that helped (eventually):** comparing lamp fixtures across photos — 3-bulb wood+black spotlight vs 4-bulb fabric-shade cylinder — but even this was a red herring because best-1 vs folder photos ALSO use different furniture stagings in the same room.
- **Fix:** Ground architecture in the folder photos (consistent across all 4 subfolders), then pick ONE furniture style (best-1 or folder) based on user preference.

### 3. Trusting blueprint dimensions over image proportions (Phase 9)
- **Symptom:** Built room-1 at 2.7m × 3.6m (from blueprint). User said "too small".
- **Root cause:** Blueprint 洋室1 dimension was 2.7 × 3.6 but the photos of the real room show a visibly bigger space — the blueprint may be for a different unit, or the room was enlarged during construction, or the photos are wide-angle and that's just how they feel to the user.
- **Fix:** User directive: **prioritize images over blueprint** for all future room sizing. Went to 3.5m × 4.0m.

### 4. Wrong reading of window positions (Phase 9)
- **Symptom:** Placed slim frosted window on north wall, curtained window on west wall (perpendicular walls). User corrected: both on same wall.
- **Root cause:** Misread the perspective in best-1. The slim frosted window appears "further back" in the image, which looked like a different wall, but it's actually further along the SAME wall (just at a different z-position than the curtained window).
- **Fix:** Both windows on west wall: slim frosted in the north half (z=1.4–1.65), curtained in south half (z=2.7–3.6).

### 5. Desk inside bed (Phase 9)
- **Symptom:** Desk geometry overlapped with bed geometry in 3D space.
- **Root cause:** I oriented the desk along z=1.8 centered, and the bed was at z=0–2.0 (head to north) — desk was literally inside the bed footprint.
- **Fix:** In the rebuild, bed at z=0.3–1.7, desk at z=2.55–3.65 along the same west wall. No overlap.

### 6. Bed facing wrong way (Phase 9)
- **Symptom:** Bed head was against the north wall, foot toward south. User said wrong.
- **Root cause:** Misread best-1 perspective — the headboard in the image is on the LEFT (west) side of the bed, not the BACK (north).
- **Fix:** Bed head against west wall, headboard is a vertical panel 0.05m thick at x=-0.02, 0.6m tall, 1.4m long along z. Bed extends EAST from headboard.

### 7. Nested-comment SyntaxError attempt (Phase 5)
- **Symptom:** When trying to preserve the old room-2 scene but disable it, I wrapped it as `const _removedOldRoom2 = ({...});` — that broke parsing because the scene body already has `});` at the end which closed the expression prematurely.
- **Fix:** Use `/* ... */` block comments instead of trying to assign a multi-statement block to a variable. For final cleanup, just delete the lines entirely.

## Architecture and design notes

### File layout pattern
- Single HTML file with inline Three.js + inline styles + inline game code
- Scene registry via `registerScene(id, cfg)` — each scene has `orbitPos`, `orbitTarget`, `fpStart`, `fpYaw`, `bg` color, `rooms[]` for the room-label UI, `noFirstPerson` flag (exterior), and a `build(S)` function that populates `S.group` and pushes wall meshes to `S.walls[]` for collision.
- Chips in top bar switch scenes via `switchScene(id)`.
- FP walk mode (toggled by `↗ walk inside` button) uses WASD + pointer-lock mouse look.

### Coordinate conventions
- +x = east, -x = west, +z = south, -z = north, +y = up
- fpYaw=0 faces -z (north); direction vector `(-sin(yaw), 0, -cos(yaw))`
- Wall constant: `WT` (wall thickness) = 0.105m
- Ceiling height: 2.4m (consistent across rooms)
